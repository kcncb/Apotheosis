# Adaptive Reactive Tracker (ART) — 设计文档

## 一、问题定义

为三角洲行动（Delta Force）的 YOLO 双机自瞄系统设计统一的移动+预测算法，替代当前 Kalman + AimController + Bezier 三件套。

### 当前系统的问题

| 问题 | 原因 |
|------|------|
| 静止目标抖动 | Kalman 的 Q/R 配比难以同时满足静止平滑和运动响应 |
| AD 急停超调 | Kalman 假设匀速/匀加速动力学，零惯性游戏每次变向都超调 |
| 参数太多 | Kalman 6 个 + AimController 4 个 + Bezier 6 个 = 16 个参数 |
| 远近目标不统一 | 远处小目标噪声大抖动，近处大目标没这个问题 |
| 切目标抖动 | Kalman 状态残留，新旧目标间来回振荡 |
| 倍镜需重调 | 参数和画面比例耦合 |

### 设计要求

1. 静止目标不抖，开始移动时快速锁死
2. 运动目标准星紧跟（不考虑弹道飞行时间）
3. 适配三角洲所有身法：滑铲、喷气推进、AD 急停、jiggle peek、袋鼠跳、拜年趴下
4. 远近目标（大小 bbox）都不抖且紧跟
5. 一套 speed_x/y 适配所有倍率瞄准镜
6. 调参简单，参数内化动态变化
7. 2D 画面空间设计
8. 多目标切换无抖动
9. 适配 60/120/144/200/240 FPS
10. 移动和预测写在一套算法里

## 二、三角洲行动运动特性分析

### 核心发现：近零惯性系统

三角洲行动的移动系统接近**零惯性** — 从满速到停止、方向反转几乎在 1 帧内完成，接近阶跃函数。这是算法设计最关键的约束，直接否决了基于动力学模型的方案（Kalman、Alpha-Beta 等）。

### 身法分类

| 身法 | 特征 | 可预测性 |
|------|------|---------|
| 匀速冲刺 | 持续线性移动 | 高 |
| AD 急停 | 瞬时反向，每次约 60 cm | 极低 — 阶跃 |
| Jiggle Peek | Q/E 探头，5-7 Hz 振荡，暴露减少 60-70% | 中 — 周期性 |
| 红狼滑铲 | 约 5m 距离，可跳取消，剪刀跳双重变向 | 中 |
| 威龙喷气推进 | V 键 10m 瞬移，15s CD | 不可预测 |
| 拜年趴下 | 瞬间卧倒，Y 轴骤降 | 低 |
| 跳射 | 精度惩罚极小，抛物线轨迹 | 中 |

注：具体速度数值（冲刺 m/s、滑铲距离等）来自社区测试，不完全确定，但算法不依赖这些具体数值 — 所有适应都基于实时检测到的像素速度和 bbox 大小。

### 关键结论

1. **不能用动力学模型做预测** — 零惯性系统违反 Kalman 的基本假设
2. **应该做滤波而非预测** — 最优估计"目标现在在哪"，而非"下一帧在哪"
3. **唯一需要的预测是补偿滤波器自身的延迟**

## 三、算法选型：为什么是 1€ Filter

学术来源：Casiez et al., "1€ Filter: A Simple Speed-based Low-pass Filter for Noisy Input in Interactive Systems", CHI 2012

| 方案 | 适合度 | 原因 |
|------|--------|------|
| Kalman Filter | ✗ | 假设匀速/匀加速动力学，零惯性下每次变向超调 |
| Extended Kalman | ✗ | 同上，复杂度更高 |
| IMM (多模型交互) | △ | 理论优雅但实现复杂，需手工定义模型转移矩阵 |
| Particle Filter | ✗ | 2D 点追踪杀鸡用牛刀 |
| Moving Average | ✗ | 固定延迟，无法自适应 |
| Alpha-Beta Filter | △ | 简化 Kalman，同样有动力学假设 |
| **1€ Filter** | **✓** | 专为交互式 2D 追踪设计，无动力学假设，jitter-lag 最优 |

核心思路：**用目标移动速度来决定平滑多重**

- 不动 → 使劲平滑 → 不抖
- 在动 → 放轻平滑 → 跟得上

### 相比原始 1€ 的改进

| 改进 | 解决的问题 |
|------|-----------|
| 双速率速度估计 (慢+快) | 运动起步响应慢 |
| 方向一致性检测 | AD 急停/振荡时预测超调 |
| BBox 自适应噪声底 | 远近目标抖动差异 |
| 突变检测 + target_id 重置 | 瞬移/切目标抖动 |
| 自适应预测时间 | 固定预测值无法适配所有速度 |
| 集成比例驱动 | 统一移动+预测，减少参数 |
| 重复帧检测 | YOLO 帧率 > 游戏帧率时速度虚高 |
| BBox 对角线平滑 | bbox 帧间抖动 ±5px 传导到截频 |

## 四、算法详细设计

### 4.1 坐标空间

所有计算在**检测像素空间**（detection pixels）进行，不做归一化。

原因：新代码采用**中心裁切**（不是缩放），1 检测像素 = 1 屏幕像素。换检测分辨率（裁切大小）不会改变同一目标的像素坐标，只会改变能看到的范围。因此无需除以 detection_res。

内部像素常数要么固定（差异可忽略），要么跟着 bbox 对角线自动适配，不需要知道屏幕分辨率。

### 4.2 整体流程

```
输入: pivot(x,y)  crosshair(x,y)  dt  bbox(w,h)  target_id
  │
  ├─ 1. dt 限幅: clamp(dt, 0.001, 0.1)
  │
  ├─ 2. target_id 变化 → 清空所有状态，从零开始
  │
  ├─ 3. 突变检测: |测量 - 滤波位置| > 1.15×bbox对角线 → 硬重置
  │
  ├─ 4. 首帧: 记住位置，直接算鼠标移动 → 跳到第 10 步
  │
  ├─ 5. 原始速度: (这帧位置 - 上帧位置) / dt
  │     如果 dt < 3ms → 跳过速度更新（重复帧保护）
  │
  ├─ 6. 双速率速度滤波
  │     ├─ v_slow: EMA @ 1.5 Hz → 稳定，控制截频
  │     └─ v_fast: EMA @ 10 Hz  → 灵敏，用于预测和方向检测
  │
  ├─ 7. BBox 自适应截频
  │     ├─ bbox 对角线做 EMA 平滑 @ 1.5 Hz
  │     ├─ 和参考值(70px)比 → 算出基础截频
  │     └─ 加上 β × |v_slow| → 最终截频
  │
  ├─ 8. 位置滤波: EMA @ 最终截频
  │
  ├─ 9. 方向一致性 + 预测
  │     ├─ 快慢速度方向一致 → 放心预测
  │     ├─ 方向相反（刚变向）→ 停止预测
  │     └─ 预测量 = 快速度 × 自适应预测时间
  │
  └─ 10. 比例驱动
        ├─ 误差 = 瞄准点 - 准星
        ├─ 鼠标移动 = 灵敏度 × 误差 × lock增益
        └─ 亚像素残余累积到下帧
```

### 4.3 重复帧保护（第 5 步）

YOLO 帧率 > 游戏帧率时，部分 step() 调用拿到的是同一张游戏画面。位置差全是噪声，但 dt 很小（如 3-5ms），除出来速度虚高。

处理：dt < 3ms 时只更新位置滤波（位置几乎没变不影响），跳过速度更新，避免噪声被小 dt 放大。

### 4.4 双速率速度估计（第 6 步）

同一个原始速度过两个不同迟钝度的滤波器：

**慢速度 v_slow (1.5 Hz)**：约 130ms 跟上真实速度的 90%。非常稳定，不被噪声骗。用来控制平滑强度。

**快速度 v_fast (10 Hz)**：约 25ms 跟上。用来做预测和方向检测。

目标突然从静止开始跑时：
- 25ms 内 v_fast 已经知道目标在动了 → 预测开始补偿
- 130ms 内 v_slow 还以为目标没怎么动 → 平滑还是偏重
- v_fast 的预测补偿了 v_slow 反应慢造成的额外滞后
- 130ms 后 v_slow 追上来，平滑自动变轻，一切归位

### 4.5 BBox 自适应噪声底（第 7 步）

YOLO 质心噪声约 ±2 像素，不管目标多大。对 bbox 170 的近处大人是 1.2%，对 bbox 36 的远处小人是 5.6%。小目标需要更重的平滑。

做法：算 bbox 对角线，和参考值比较。

bbox 对角线本身帧间也会跳（约 ±5 像素），所以先做一次 EMA 平滑（1.5 Hz），再拿平滑后的值算基础截频。

参考值 70 像素对应约 30×60 的 bbox，即中距离站立人形目标。

FOV 透视畸变的影响：同一目标在屏幕边缘比中心看起来略大（10-20%），但第一帧直接 snap 到目标不走这个逻辑，第二帧起准星已在拉向目标，几帧后目标接近画面中心，差异消失。FOV 70-120 范围内影响可忽略。

### 4.6 方向一致性（第 9 步）

预测的目的是补偿滤波延迟。但 AD 急停时如果还在预测，准星会冲过头。

判断方法：看 v_fast 和 v_slow 是不是指向同一个方向。

- **匀速前进**：两者同向 → 放心预测
- **刚刚急停**：v_fast 已转向（反应快），v_slow 还指旧方向（反应慢）→ 方向相反 → 停止预测
- **静止**：两者速度都很小，全是噪声 → 不预测

急停后约 130ms v_slow 追上新方向，一致性恢复，预测恢复。自然形成"变向→暂停预测→确认方向→恢复预测"的行为。

蹲下场景：Y 轴瞬间位移约 60 像素（中近距离），不触发 snap（中距离阈值 ≈ 1.15×70 = 80），走正常滤波。v_fast 瞬间出现大的向下分量，v_slow 还接近零 → 一致性低 → 预测被压制 → 不会向下过冲。

### 4.7 动态预测时间（第 9 步）

不是固定值，而是从滤波器截频反推：

- 平滑越重 → 滤波延迟越大 → 需要补偿越多
- 平滑越轻 → 延迟本来就小 → 补偿少一点

取滤波器群延迟的一半作为预测时间（完全补偿会在变向时过冲），再乘以方向一致性和 warmup（新目标前 4 帧从 0 渐增到 1，等速度估计稳定）。

### 4.8 突变检测与目标切换（第 2、3 步）

两层保护：

**target_id 变化**：上游 tracker 切了目标，无条件清空所有状态。

**位置突变**：测量位置和滤波位置差距 > kSnapMult × 平滑后的 bbox 对角线（kSnapMult = 1.15），判定为瞬移（威龙推进）或其他异常，硬重置。

阈值跟着 bbox 走的好处：
- 自动适配分辨率 — 1440p 下目标 bbox 更大，阈值自动升高，不需要 screen_res 参数
- 自动适配距离 — 近距离大目标阈值高（近距离蹲下位移大，不会误触发），远距离小目标阈值低（更灵敏地检测异常跳变）
- 典型值：中距离 bbox 对角线 70 → 阈值 80，和之前固定值一致
- 首帧 sd 未初始化时用 70 作为 fallback

### 4.9 比例驱动（第 10 步）

误差 = 瞄准点 - 准星位置（准星通过找色动态定位，不是固定画面中心）。

鼠标移动 = 灵敏度 × 误差。鼠标只能移整数像素，小数部分累积到下帧（亚像素累积），确保微小持续误差最终也能修正。

Lock boost：准星离目标越近增益越大，有"粘住"手感。默认开启。

## 五、用户参数

```
speed_x = 0.6           鼠标灵敏度 X
speed_y = 0.6           鼠标灵敏度 Y
lock_strength = 0.5     吸附强度 [0,1]
lock_radius_px = 25.0   吸附半径 (px)
```

**调参方法**：对准一个静止目标，调 speed_x/y 直到准星一次到位。其余全自动。

不同倍镜下 speed_x/y 不需要单独调。原理：开镜后采集的是放大画面，bbox 自然变大，算法自动适配。游戏内置的倍镜灵敏度会影响收敛速度（快几帧或慢几帧），但不影响准确性，不会震荡。

## 六、内部常数

所有常数固定，不需要任何分辨率参数。突变检测阈值通过 bbox 对角线倍数自适应，其余像素常数固定（1080p/1440p 差异不到 10%，不可感知）。

| 常数 | 值 | 含义 |
|------|------|------|
| kFcBase | 1.0 Hz | 基础位置截频 |
| kBeta | 0.022 | 速度→截频系数 (px/s → Hz) |
| kFcSlow | 1.5 Hz | 慢速度滤波截频 |
| kFcFast | 10.0 Hz | 快速度滤波截频 |
| kRefDiagPx | 70.0 px | 参考 bbox 对角线（固定） |
| kSnapMult | 1.15 | 突变检测 = kSnapMult × 平滑 bbox 对角线 |
| kConsThreshPx | 80.0 px/s | 方向一致性速度门槛（固定） |

kSnapMult 替代了原来的固定 kSnapPx — 阈值跟着 bbox 走，自动适配目标距离和屏幕分辨率。kRefDiagPx 和 kConsThreshPx 保持固定值，1080p/1440p 之间差异极小（fc_min 偏差 ~30%、速度门槛噪声影响 <10%），体感上无法区分。

## 七、接口设计

### 调用接口

```cpp
ArtResult step(
    double pivot_x, double pivot_y,    // 目标瞄准点 (检测像素)
    double cross_x, double cross_y,    // 准星位置 (检测像素，找色定位)
    double dt,                          // 帧间隔 (秒)
    int bbox_w, int bbox_h,            // 检测 bbox 尺寸
    int target_id = -1                  // 锁定目标 ID
);
```

### 输出

```cpp
struct ArtResult {
    int move_x, move_y;     // 鼠标增量 → 直接发给硬件驱动
    double aim_x, aim_y;    // 瞄准点 (诊断用)
    double vel_x, vel_y;    // 速度估计 (诊断用)
    double cutoff_hz;       // 当前截频 (诊断用)
    double consistency;     // 方向一致性 (诊断用)
    bool snapped;           // 是否触发了突变重置
};
```

### 内部状态

```
fx, fy        滤波后位置 (px)
dsx, dsy      慢速度 (px/s)
dfx, dfy      快速度 (px/s)
pmx, pmy      上帧测量 (px)
ssp           标量速度 EMA (px/s，预留振荡检测)
sd            平滑后的 bbox 对角线 (px)
rx, ry        亚像素残余 (鼠标像素)
age           帧计数
tid           上一个 target_id
init          是否已初始化
```

### 集成路径

替换 mouse_thread_loop.cpp 中的调用链：

```cpp
// ── 之前 ──
kalman_.update(pivotX, pivotY, dt, lookahead);
auto [mx, my] = aimController_.step(errX, errY, det_width);

// ── 之后 ──
auto r = tracker_.step(pivotX, pivotY, crosshairX, crosshairY,
                       dt, bbox_w, bbox_h, target_id);
queueMove(r.move_x, r.move_y);
```

## 八、游戏细节对算法设计的影响

以下是三角洲行动的具体特性如何影响了算法每个设计决策。

### 8.1 零惯性移动 → 否决 Kalman，选用 1€ Filter

三角洲的角色移动没有加速/减速过渡，方向反转在 1 帧内完成。Kalman 滤波器假设目标做匀速或匀加速运动，在每次急停/变向时都会预测错方向并超调。1€ Filter 不做任何动力学假设，只看"目标动多快就信多少"，天然适合零惯性系统。

### 8.2 AD 急停/Jiggle Peek → 方向一致性检测

AD 急停和 jiggle peek 的共同特征是频繁变向。如果不处理，每次变向时预测都会把准星推向错误方向然后再拉回来，表现为抖动。

方向一致性（v_fast 和 v_slow 的方向比较）在变向瞬间自动归零，停止预测，等新方向确认后恢复。这对 AD 频率（~2-3 Hz）和 jiggle peek 频率（5-7 Hz）都有效。

### 8.3 威龙 10m 瞬移 → 突变检测（bbox 倍数自适应）

威龙的 V 键推进在 1-3 帧内完成 10 米位移，在画面上表现为几百像素的瞬移。突变阈值 = 1.15 × bbox 对角线（中距离约 80px），远小于瞬移距离，可以干净地检测。

阈值跟着 bbox 走而非固定值，额外的好处：近距离大目标阈值自动升高（bbox 对角线 140 → 阈值 161），不会因大幅度正常动作误触发 snap；远距离小目标阈值自动降低（bbox 对角线 30 → 阈值 35），对异常跳变更灵敏。同时自动适配 1080p/1440p — 同一目标在 1440p 上 bbox 更大，阈值跟着变大，不需要 screen_res 参数。

### 8.4 蹲下约 60px 位移 → kSnapMult 1.15 保证不误触发

中近距离蹲下 Y 轴约 60 像素位移。中近距离 bbox 对角线约 70 → 阈值 = 1.15 × 70 = 80，蹲下不触发 snap，走正常滤波 — 此时方向一致性机制会压制预测，防止 Y 轴过冲。

### 8.5 交战距离 0-80m → bbox 参考值 70px

三角洲多数战斗在 0-80 米。中距离（约 30-50 米）的人形目标在 1080p 下 bbox 约 30×60 像素（对角线 ~70）。以此为参考值，近处目标（对角线 > 70）自动减轻平滑提高响应，远处目标（对角线 < 70）自动加重平滑压制噪声。

### 8.6 中心裁切采集 → 不做归一化

新代码采集方式是中心裁切（不是将全画面缩放到检测分辨率）。裁切模式下 1 检测像素 = 1 屏幕像素，换检测分辨率不改变坐标比例，只改变视野范围。因此算法直接在像素空间计算，无需归一化。

### 8.7 开镜采集放大画面 → bbox 自适应自动生效

开 4 倍镜后采集的是放大后的画面，目标 bbox 自然变大。bbox 自适应机制自动减轻平滑、提高响应。无需为不同倍镜单独调参。

不同倍镜下游戏内置灵敏度不同，会影响 speed_x/y 的收敛速度（快几帧或慢几帧），但不影响准确性，不需要单独调。

### 8.8 准星找色定位 → cross_x/y 是动态值

游戏有后坐力，准星（红点）会随射击跳动。新代码用找色功能实时定位准星，cross_x/y 是动态的真实准星位置，不是固定的画面中心。算法直接用这个动态准星计算误差，自然处理了后坐力带来的准星偏移。

### 8.9 YOLO 帧率 > 游戏帧率 → 重复帧保护

YOLO 推理速度通常高于游戏帧率，部分 step() 调用拿到的是同一张游戏画面。两次检测位置几乎一样，但 dt 很小（3-5ms），除出来速度虚高（噪声/小dt = 大值）。

解法：dt < 3ms 时跳过速度更新，只更新位置滤波。位置几乎没变所以滤波结果不受影响，但避免了速度被污染。

### 8.10 bbox 帧间抖动 ±5px → bbox 对角线平滑

YOLO 的 bbox 在连续帧间大小会跳约 ±5 像素。如果直接用来算基础截频，fc_min 会跟着跳 ±10%。对 bbox 对角线做一次 EMA 平滑（1.5 Hz）再使用，消除这个传导。

### 8.11 屏幕分辨率 1080p/1440p → 无需额外参数

1440p 下同一目标占更多像素，bbox 更大。突变检测阈值（kSnapMult × bbox 对角线）自动跟着 bbox 变大，天然适配。kRefDiagPx（70）和 kConsThreshPx（80）保持固定，1440p 下 fc_min 偏高约 30%（平滑略轻）、速度门槛相对略低，但这两项差异在实际操作中不可感知 — 完全消除了 screen_res 参数。

### 8.12 FOV 70-120 → 影响可忽略

不同 FOV 下屏幕边缘的透视畸变不同（FOV 120 最大，边缘 bbox 比中心大约 20-30%）。但 bbox 自适应只影响平滑强度不影响拉枪速度，且第一帧直接 snap 不走 bbox 逻辑，后续帧随着准星拉向目标，目标逐帧靠近画面中心，差异消失。

## 九、已知局限与未来改进方向

### 已知局限

1. **中速线性运动有 2-5 px 残余滞后**：预测补偿了大部分但不完全。对于身体目标可接受，极端头部追踪可能需要更激进的预测。

2. **振荡检测可以更精确**：当前通过方向一致性间接检测振荡。可以加入 scalar_speed vs net_speed 比值（ssp 状态已预留）来直接检测 jiggle peek，进一步减少振荡时的预测超调。

3. **小目标牺牲了一些响应性**：bbox 越小截频越低。这是抖动 vs 响应的基本权衡。

4. **不同倍镜下收敛速度有差异**：一套 speed_x/y 在不同倍镜下收敛快慢不同（因游戏灵敏度不同），但不影响准确性。

### 未来改进

- **振荡检测层**：scalar_speed / net_speed 比值 → 直接识别 jiggle peek 并进一步抑制预测
- **检测延迟补偿**：将推理耗时传入 step()，加到预测时间上
- **置信度加权**：YOLO 置信度越低 → 加重平滑
