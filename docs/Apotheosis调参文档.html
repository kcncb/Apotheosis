这份文档怎么用

Apotheosis 是一个把"识别 → 跟踪 → 控制 → 反馈"四段流水线串起来的实时系统。每个模块都至少有 3 个相互耦合的旋钮，从零盲调几乎一定走偏。这份文档的目标是：

  • 每个参数都讲三件事：它是什么 · 它为什么存在 · 怎么影响表现。
  • 给一份从零到能用的顺序——你按这个顺序调，每一步只动一组参数，前一步不稳后一步不动。
  • 常见坑写在每一节末尾的红框里。
第一次接触强烈建议先读 系统心智模型 和 推荐调参顺序，再跳到具体模块。

系统的整体心智模型

把整个 ai.exe 想象成五条独立的线程，通过几个共享原子量串起来：

  1. 采集线程：把屏幕（或网络流）抓成 detection_resolution × detection_resolution 的方块图。
  2. 检测线程：YOLO（DML 或 TRT 后端）跑一帧，输出 boxes + class_ids。可选叠一层「深度遮罩抑制」。
  3. 跟踪线程：MultiTargetTracker 把这些 box 关联到长生命周期的 track，挑一个作为 locked target。
  4. 鼠标线程：根据 locked target 的 pivot 跑 PID + Kalman，输出鼠标 dx/dy；同时跑智能扳机判断是否开枪。
  5. 叠层线程：渲染所有调试信息（不参与控制循环）。

关键观察：检测的"瞄准锚点"（pivot）和鼠标的"控制目标点"是同一个值。锚点抖一下，鼠标就抖一下；锚点跳到墙上，鼠标就把你瞄准线拉到墙上。所以——

调参的核心原则：先把 pivot 调稳（采集 + 检测 + tracker 部分），再调 PID / Kalman / 扳机。pivot 不稳的情况下调 PID 是在拟合噪声。

推荐调参顺序 (SOP)

照这个顺序走。除非上一阶段稳定，不要跳到下一阶段。

  1. 第一步：采集稳
    选好 capture 后端，确认采集 FPS不掉帧、画面无撕裂。详细 →
  2. 第二步：检测准
    调confidence_threshold到刚好不丢真目标也不出虚警；调好NMS。把无关类别先丢"删除"桶。详细 →
  3. 第三步：类别选对
    把要瞄的类别挪到"瞄准"桶，给每类设合理的Y 偏移（头≈0.15，胸≈0.45）。详细 →
  4. 第四步：FOV 设大点
    先把 FOV 开到舒适最大值（130×90 左右），不要立刻打开动态 FOV。详细 →
  5. 第五步：标定 lock_strength 上限
    speed 暂打满到 1.0，lock_radius_px 选 12（吸附锁死）或 50（动态锁死），按住热键锁移动靶慢慢加 lock_strength，屏幕刚好不抖时的最大值就是该游戏 / 该灵敏度 / 该 DPI / 该倍镜 / 该主辅机配置下永远不变的"锁死力度"。详细 →
  6. 第六步：Kalman 兜底
    开 Kalman，调reset 超时和Warmup。让目标短暂消失时鼠标不要乱跑。详细 →
  7. 第七步：回降速度
    锁死定下来之后再把 speed_x / speed_y 从 1.0 往回降，控制器手感"刚好跟得上但不过冲"即可——一般落在 0.5 ~ 0.8 之间。开了动态锁死的话可以适当上抬到 0.7+。详细 →
  8. 第八步：增强稳定性
    开切换迟滞+Y 偏移衰减。两件事都是"近距离防抖"。
  9. 第九步：解决长得像的目标干扰
    开威胁度优先级+深度遮罩。
  10. 第十步：动态 FOV
    先把基线 FOV 调到能稳定锁主目标的最小值，再开动态 FOV，调内边距系数到 1.10~1.30。详细 →
  11. 第十一步：智能扳机
    最后一步。前面任何一项不稳，都不要开扳机——只会越打越错。详细 →

画面采集

是什么
把屏幕画面变成检测器吃的张量。
为什么有
不同采集方式（UDP / TCP / OpenCV）在延迟、稳定性、部署复杂度和外部采集设备兼容性上取舍不同。
影响
直接决定端到端延迟和掉帧率。系统其他所有部分都建立在采集稳定的前提上。
参数默认含义
capture_methodudp_capture三选一：udp_capture（外部机器 MJPEG/UDP 推流）/ tcp_capture（TCP 推流，稳定优先）/ capture_card（DirectShow 采集卡 / 虚拟摄像头）。旧的 DXGI / WinRT / OpenCV 采集入口已删除。
udp_ip / udp_port0.0.0.0 / 1234UDP 推流监听地址和端口。0.0.0.0 = 接受任意发送方。端口范围 1..65535。
tcp_ip / tcp_port0.0.0.0 / 1235TCP 推流监听地址和端口。网络波动大时比 UDP 更稳，但延迟通常更高。
capture_card_index0DirectShow 设备列表中的索引。多设备时改这个选目标采集卡。
capture_card_formatAUTO请求像素格式：AUTO / NV12 / MJPG / YUY2 / RGB32。AUTO 让设备自行选择；解码瓶颈时强制 MJPG 或 NV12 测试。
capture_card_width / height / fps0 / 0 / 0请求采集分辨率和帧率。0 = 让设备自行决定。
capture_card_crop_width / height0 / 0从采集帧中央截取的尺寸。0 = 用完整采集画面（无裁剪）。
detection_resolution320截下来后压成的方形边长。允许 32..2048，常用 160 / 320 / 640。值越大检测越准但 GPU 负担线性涨。
capture_fps60采集线程节流上限。设过高反而会让 GPU 抢占检测时间。和你显示器刷新率匹配即可。
circle_masktrue把方框外的圆角用黑掩遮罩。多数游戏 UI（雷达、技能图标）在四个角，开它能减虚警。

调参步骤

  1. 在游戏里跑一局，把"采集预览"窗口对着屏幕看是否流畅。有撕裂、黑屏或卡顿优先换采集后端。
  2. 如果用 udp_capture 或 tcp_capture，先把 IP / 端口对好——进游戏看不到画面通常就是这里的问题。两者均支持 GPU JPEG 解码（在 CUDA 构建上自动启用）。
  3. 如果用 capture_card：从 capture_card_index = 0 起，format = AUTO，width/height/fps = 0 让设备自报；不出图就强制 MJPG 或 NV12 重试。需要中央裁剪时填 capture_card_crop_width/height。
  4. detection_resolution 默认 320 适合大多数情况。瞄远距离或小目标提到 640；追求最低延迟降到 160。
切换 detection_resolution 后 tracker 会自动 reset。所以如果你正在调 Kalman 千万别中途改这个值，否则之前调的全白调。

检测分辨率与阈值

参数默认含义
backendTRTTRT 比 DML 快 2–4 倍但只能在 NVIDIA。无 CUDA 自动 fallback 到 DML。
ai_modelsunxds_0.5.6.engineYOLO 引擎文件。每个模型有自己的类别 id——换模型后必须重新做"类别分桶"。
confidence_threshold0.10低于这个置信度的检测直接丢。这是降虚警的第一道闸。
nms_threshold0.50非极大值抑制 IoU 阈值。值越小同一个目标越不容易被拆成两个框，但贴在一起的两人越容易被合并成一个。
max_detections100每帧最多输出多少个 box。够用就行，过大白白吃 NMS 算力。
dml_device_id0DML 后端使用的显卡序号。多显卡机器可用它指定 DirectML 设备。
export_enable_fp8false导出 TensorRT engine 时是否尝试 FP8。仅在显卡、TensorRT 和模型都支持时才有意义。
export_enable_fp16true导出 TensorRT engine 时启用 FP16。大多数 NVIDIA 显卡都建议保持开启。
fixed_input_sizefalse导出/推理时是否固定输入尺寸。需要和模型、engine 构建方式一致。
use_cuda_graphfalseTRT 路径的 CUDA Graph 优化开关。可降低调度开销，但和双缓冲流水线二选一。
use_double_bufferfalse双缓冲流水线：让第 N 帧 CPU 后处理与第 N+1 帧 GPU 预处理/推理重叠。吞吐更好，但会换来约 1 帧延迟。
use_pinned_memoryfalse使用页锁定内存加速 CPU↔GPU 传输。内存很紧或驱动异常时保持关闭；要压榨传输性能时再单独测试打开。
gpuMemoryReserveMB2048给系统和游戏预留的显存预算，避免推理把显存吃满导致游戏掉帧。
enableGpuExclusiveModetrueGPU 资源保护开关。资源紧张时优先保证游戏和系统仍有余量。
capture_use_cudatrue采集预处理尽量走 CUDA。关闭后更多工作落到 CPU，兼容性可能更好但延迟通常更高。
cpuCoreReserveCount4预留给系统和游戏的 CPU 核心数量，避免检测线程把 CPU 榨满。
systemMemoryReserveMB2048预留系统内存，防止长时间运行时因内存压力卡顿。

调参步骤

  1. 站桩看真实目标，调高 confidence_threshold 直到刚好不丢识别（通常 0.15 ~ 0.30）。
  2. 对着空场景（没人）看 1 分钟。如果偶尔出虚警，再上调 0.05；如果连真目标都丢，下调 0.05。
  3. 如果两个目标贴在一起经常被合并：把 nms_threshold 从 0.50 降到 0.40。
  4. TRT 后端先保持 use_cuda_graph=false 和 use_double_buffer=false 调准功能；稳定后再单独测试性能开关，两个不要同时开。

类别分桶（删除 / 过滤 / 瞄准）

是什么
每个模型类别可被分到三个桶。
为什么有
同一个模型可能输出 80 个类别（COCO），其中只有 person/head 真正有意义。
影响
"删除"=NMS 后立刻丢，下游完全不知道；"过滤"=保留可见但不进瞄准候选池；"瞄准"=可被锁定。
错误用法
把 chair / dog / bottle 留在"过滤"桶，本以为不影响——但 tracker 仍然花时间维护它们的轨迹。
正确用法
不需要的类别一律设成"删除"。"过滤"只用于 debug 期间想看见检测框但暂时不想瞄的类别。
桶配置值进入哪些流程
删除DeleteNMS 后立刻丢弃，不进 tracker、不显示、不参与瞄准。
过滤Filter保留在检测缓冲，可用于调试显示、深度遮罩观察等，但不会进入瞄准候选池。
瞄准Aim进入 tracker 和锁定候选池，是否被某个热键使用还要看该热键的 aim_classes。
模型切换后系统会按模型元数据同步类别表：旧类别尽量保留桶设置，新出现的类别默认进"删除"。类别名优先来自 ONNX 元数据，其次读取同名 .json / .names 旁路文件，最后退回 class_N。

按键绑定

Apotheosis 只允许鼠标四键触发瞄准：左/右/侧键 X1/X2。键盘键和数字键全部不接受——这是为了防止误触发以及兼容硬件输入设备。

可以创建多个热键，按下时列表里第一个匹配的生效。同时按多个会按列表顺序选。所以"主瞄"放上面、"仅头瞄"放下面是常见配置。

瞄准类别 + Y 偏移

参数默认含义
aim_classes[]本热键关心的类别列表，顺序就是优先级。
y_offset0.50每类独立。0=瞄 bbox 顶（头）、0.5=中心（胸）、1=底（脚）。
aim_class_kalman_N关闭每个瞄准类别可独立覆盖 Kalman 的位置噪声、速度噪声、测量噪声、速度衰减和最大速度。用于同一热键同时瞄 head/body 时避免一套参数两边都不合适。

典型配置

  • 头部专精：head: y=0.10 单独一项。
  • 主瞄混合：head: y=0.15（优先一）+ person: y=0.40（优先二，胸口偏上）。
Y 偏移设 0 容易瞄到头顶帽子上方的空位（bbox 通常会包到帽子）。头瞄留 0.10 ~ 0.20 的余量更稳。

视野 / FOV / 动态 FOV

是什么
"瞄准区域"——准星周围多大范围内的检测才算候选。
为什么有
没有 FOV，所有屏幕上的目标都会被认作候选，背后转身的人会被优先锁。
影响
FOV 大 → 反应灵活但易误锁；FOV 小 → 钉死主目标但闪不过去。
参数默认含义
FOV X / FOV Y106 / 74椭圆区域的直径，单位是检测分辨率像素。
例如 detection_resolution=320，FOV X=106 → 区域横向占屏宽 106/320 ≈ 1/3。
dynamic_fov_enabledoff开启后 FOV 按瞄准距离动态收缩。
dynamic_fov_margin_frac1.10收紧时 = bbox × 这个倍数。1.10 ≈ "正好框住识别框留 10% 余量"。
dynamic_fov_min_radius_frac0.20收缩下限 = 基线 FOV × 这个比例。防止极小目标导致 FOV 几乎归零。

动态 FOV 是怎么工作的

每帧（在 tracker 跑之前）：

  1. 取上一帧锁定目标的 pivot 与 bbox。
  2. 算 err = |pivot - 准星|。
  3. shrink_alpha = clamp(1 - err / 基线半径, 0, 1)。
    • err = 0（瞄准） → alpha=1 → 收最紧
    • err ≥ 基线半径（边缘） → alpha=0 → 用基线最大值
  4. 有效 FOV = lerp(基线, bbox × margin, alpha)。

调参步骤

  1. 关掉动态 FOV。把 FOV X/Y 调到能舒适锁住主目标的最小值（不是最大！）。常见 80×60 ~ 130×90。
  2. 能稳定锁住目标后，开动态 FOV。
  3. margin_frac 默认 1.10。跟枪经常在收缩瞬间丢锁就提到 1.20、1.30。
  4. min_radius_frac 一般不动。如果常打远距离小目标，可以提到 0.30。
  5. 在调试面板「动态 FOV (实时)」看实时数值，确认收缩在合理范围（不要看到 5×5 这种极小数）。

速度增益（speed_x / speed_y / lock_strength）

是什么
把"目标 - 准星 = 误差"映射成"鼠标 dx/dy"的三参数比例控制器。
移动量 = speed × err × mult。
为什么有
早期版本是 PID + Flick/Track 双模式，参数互相耦合且难调。现在简化为：远距比例增益 + 近距吸附两个旋钮，没有积分、没有微分、没有 per-tick 钳位——大移动一次推到位，色控枪 / 后座补偿不被限速。
影响
speed 太大 → 过冲 / 振荡；太小 → 跟不上目标。lock_strength 控制最后一两像素的"咬死"，独立于 speed。
参数默认含义
speed_x0.6横向比例增益，钳位 [0, 1]。每帧 dx = speed_x × err_x × mult。
speed_y0.6纵向比例增益，钳位 [0, 1]。一般和 X 接近，但有些游戏 Y 灵敏度需要单独调小。
lock_strength0.0近距吸附力度，钳位 [0, 1]。= 0 时短路整个动态锁死分支，回退成纯 P 控制（用于先标定 speed 基线）。详见吸附锁死。
lock_radius_px25吸附 / 动态锁死的生效半径（检测像素），钳位 [0, 200]。0 = 用旧的 0.08×检测分辨率。

核心公式：

move_x = speed_x × (target_x - cursor_x) × mult
mult   = 1 + 4 × lock_strength × atten × R² / (R² + err²)   (lock_strength > 0)
mult   = 1                                                  (lock_strength = 0, 短路)
R      = lock_radius_px                                     (用户可调, 0 时回退 0.08×detection)
atten  = 1.0  (Path A：新检测帧)
       = 0.65 (Path B：准星找色 / Kalman 外推 → 后座补推)
没有 PID，没有 I/D，没有 Flick/Track。如果你看到老资料提到 pid_p / pid_d / flick_track_enabled / pid_track_p 等键名，那是已经移除的旧实现。

调参步骤（按外部资料的"先标锁死再调速度"流程）

  1. 速度先打满：speed_x = speed_y = 1.0（新钳位上限），lock_strength = 0.1。
  2. 进训练场，离目标近一些，按住热键锁移动靶（斜上 / 斜下扫过去）。
  3. 慢慢加大 lock_strength。随着这个值变大，屏幕开始抖——屏幕刚好不抖时的最大值就是你的锁死力度上限。
  4. 这个 lock_strength 在同游戏 / 同灵敏度 / 同 DPI / 同倍镜 / 同主辅机配置下永远不变；以上任意一项变了就要重新标。
  5. 锁死定下来之后再调 speed：speed_x / speed_y 从 0 ~ 1 之间挑（一般 0.5 起步，跟不上 +0.1）。
开了动态锁死后第 3 步的"不抖临界"会偏低（动态锁死本身已经在近距放大），通常比纯吸附锁死再降 0.05 ~ 0.10。具体见 吸附锁死。
站桩调出来的 speed 在实战中容易过冲——动目标的误差变化率高得多。一定要在动目标上验。如果实战明显过冲，speed 回退一档，依赖 Kalman 的预测把跟枪的"超前量"补回来，而不是靠加大增益。

Kalman 兜底 + 类别覆盖

是什么
基于运动学（位置 + 速度）做"目标短暂消失时下一帧在哪"的预测。
为什么有
检测会偶尔丢一两帧（被烟雾、UI 遮挡），如果 PID 直接停止鼠标会"卡一下"。
影响
参数错了会把鼠标"甩飞"——丢检测的瞬间 Kalman 按速度外推，预测错了鼠标就跟着跑。
参数默认含义
kalman_enabledtrue总开关。关掉则丢检测瞬间鼠标立刻停。
predictionInterval0.01 s无条件向前预测的基础窗口（秒）。最终 lookahead = predictionInterval + (检测延迟 if compensate_detection_delay) + additional_prediction_ms。0 = 仅由后两项决定,完全没有"主动超前量"。
prediction_futurePositions20调试叠层里画几个未来点（仅可视化,不进控制器）。
draw_futurePositionstrue是否画未来点轨迹。
process_noise_position40"模型相信目标位置变化的不确定度"。大 → 更信测量、不信预测。
process_noise_velocity1800同上但对速度。大 → 允许速度突变（适合无规律行人）。
measurement_noise35"模型相信测量本身有多吵"。大 → 平滑但延迟；小 → 灵敏但抖。
velocity_damping0.08速度自衰减系数。无观测时速度按 (1-d) 每帧衰减，防止外推飞出去。
max_velocity20000速度硬上限（px/秒）。防止误检测产生瞬时跳变让模型估出疯狂速度。
warmup_frames2前 N 帧只观测不外推。新锁目标时给模型几帧"热身"。
compensate_detection_delaytrue检测有 ~1 帧延迟，开它会向前外推一点补偿。
additional_prediction_ms0额外向前预测多少毫秒。负数 = 故意滞后。
reset_timeout_sec0.50多久没看到目标就放弃 Kalman 预测。设大鼠标会被"幽灵"拖很远。

每个 aim class 还能单独覆盖

头部和身体的运动学很不一样：身体走直线慢，头一甩就 90 度。所以每个 HotkeyAimClass 可以独立打开"Kalman 覆盖"，给 head 一组更"信任测量、允许更高速度"的参数：

  • head 推荐：process_noise_position=30, process_noise_velocity=2400, measurement_noise=18, max_velocity=30000。
  • body 用默认即可。
切换到不同类别的目标时，Kalman 状态会自动 reset（因为参数变了）。不必担心"head 学的快速运动会污染 body 的预测"。

调参步骤

  1. 关掉双模式 PID 干扰，先单独看 Kalman。
  2. warmup_frames 设 2-3，让 Kalman 不在第一帧就乱预测。
  3. 故意让目标短暂走到掩体后看鼠标行为：原地等回来=参数偏保守；追到掩体后=参数偏激进。
  4. reset_timeout_sec 0.3 ~ 0.6 之间。0.5 一般够。
  5. 看到鼠标在目标重新出现的瞬间往反方向猛甩——降 velocity_damping 到 0.15+ 帮速度更快归零。

吸附锁死 / 动态锁死 (lock_strength + lock_radius_px)

是什么
目标越靠近准星，比例增益就被额外放大——产生"磁吸"效果。lock_radius_px 决定"靠近"的判定半径。
为什么有
纯 speed × err 在小误差时输出极小，最后一两像素总在准星周围磨蹭。给个距离衰减的乘子解决。
影响
lock_strength 数值 [0, 1]，0 关闭并短路整个分支；0.3~0.5 是温和磁吸，0.8+ 明显感觉"鼠标自己动"。lock_radius_px 决定 R 大小，外部资料里的"锁死范围 8~16"和"动态锁死阈值 40~65"指的就是这个。

公式：

mult = 1 + 4 × lock_strength × atten × R² / (R² + err²)
R    = lock_radius_px            (用户调；默认 25)
atten = 1.0   Path A (新检测帧)
       0.65  Path B (准星找色 / Kalman 外推 → 后座补推)

解读：err = 0（咬住目标）→ mult = 1 + 4 × lock × atten，lock=1 atten=1 时增益变 5×；err = R 时 mult = 1 + 2 × lock × atten；err ≫ R 时 mult → 1 吸附无效。R 越小吸附区越窄、越靠近准星才放大；越大整张图都偏锁，开高 lock 更易抖。

两种风格对照

风格lock_radius_pxlock_strength 起点speed_x / speed_y 起点说明
吸附锁死（近距咬死）8 ~ 160.300.50 / 0.50R 小，只有最后一两像素吸；远距完全靠 speed。
动态锁死（中远距过渡）40 ~ 650.20 ~ 0.300.65 ~ 0.75R 大，吸附覆盖整个目标核心。开了之后 lock 要降一档否则抖；同时 speed 可适当上抬。

近点跳拉抖动的解决（动态锁死专用）

现象
把动态锁死开起来后，近距打跳枪 / 拉枪，准星会肉眼可见地抖。
原因
大 R 让吸附区覆盖了整个目标，准星在跳拉过程中频繁穿越 R，乘子在 1 ~ (1+4·lock) 之间剧烈切换。
解法
  1. 把原来标好的 lock_strength（比如 0.35）降一档到 0.25，到不抖为止。
  2. 同时把 speed_x / speed_y 从 0.5 抬到 0.7 ~ 0.75，把"远点跟枪能力"补回来。
  3. lock_radius_px 控在 40 ~ 65，越小越不容易触发；越大触发更频繁但吸附区更大。
Path B 自动衰减 0.65×：准星找色 / Kalman 外推这两条路径在控制器里会自动把 lock 系数打 0.65 折，专门用来抑制后座位移被锁死放大成"准星抖"。不需要为 Path B 单独调任何参数，它对 Path A 的标定值是透明的。

Bezier 轨迹模式 (可选)

是什么
把"准星 → 目标"的直线段重塑成一条三次贝塞尔曲线,把曲线偏置后的等效误差喂给同一份 P 控制器。
为什么有
让鼠标轨迹在视觉上更像人手挥动,而不是机器人式直线。仅改变路径形状,最终目标点仍是原始 target。
影响
速度 / 锁定 / Kalman 全部不受影响,Bezier 只在 driveAimToTarget 入口处替换 err 向量。Direct 模式下后续字段全部静默忽略。
参数默认含义
aim_trajectory_modeDirectDirect = 现有非线性 P 控制器（直线靠近）。Bezier = 启用本节后续字段。
bezier_cx1, bezier_cy10.30, 0.25第一控制点。归一化坐标:X 沿起→终方向,Y 垂直方向,单位 = 起终段长度。
bezier_cx2, bezier_cy20.70, -0.15第二控制点。两控制点 Y 异号会形成 S 形轨迹（更"人味"）。
bezier_follow_alpha0.15目标移动平滑系数 [0,1]。0 = 锚定后曲线不再跟目标（横移会脱靶）;1 = 每帧完全跟（退化成直线）;中间值 ≈ 一阶低通。
bezier_reanchor_threshold_px60.0目标瞬移超此值（检测像素）时强制重锚一次新曲线,避免目标 ID 切换造成的伪轨迹。

调参步骤

  1. UI 里有可视化曲线编辑器,直接拖控制点比改数字直观。建议先把两控制点设成对称形状（如 0.3/0.2 与 0.7/-0.2）观察。
  2. follow_alpha 小（0.05~0.15）→ 曲线稳但慢横移容易脱靶;大（0.3~0.5）→ 跟得紧但曲线退化为直线。从 0.15 起步。
  3. reanchor_threshold_px 太小会在持续移动的目标上反复重锚出现锯齿;太大让锁定切换后的几帧仍走旧曲线。60 px @ detection_resolution=320 ≈ 屏宽的 1/5,通常够用。
Bezier 模式只影响轨迹形状。"咬死"和"动态锁死"两个特性仍由 lock_strength 控制,Bezier 内部把直线 err 替换成曲线偏置后的等效 err 后再交给同一份 P 控制器累积。
Bezier 是视觉伪装而非精度提升。某些反作弊关心的是"鼠标曲线是不是过于线性",Bezier 在这点上有帮助;但它会在目标快速横移时引入额外的滞后（曲线形状跟不上目标变化）。如果你追求的是精度,保持 Direct。

Y 偏移距离衰减

是什么
bbox 越大（=越近），Y 偏移越靠近 0.5。
为什么有
远处 bbox 50 px 高，y_offset=0.45 → 锚点上下浮动 ±2 px；近处 bbox 500 px 高，同 y_offset → ±20 px。近距离锚点会抖得厉害。
参数默认含义
低高度比 (low_frac)0.10bbox_h / 屏高 ≤ 这个值时不衰减（远）。
高高度比 (high_frac)0.40bbox_h / 屏高 ≥ 这个值时完全收到 0.5（近）。

目标切换迟滞

是什么
同优先级类别之间换锁时的"必须好出多少分 + 持续多少帧"门槛。
为什么有
两个目标贴在一起时，每帧的"谁更优"会因像素噪声反复切换，鼠标在两人之间来回弹。
影响
两个旋钮都越大 → 越粘 → 反应越慢。更高优先级类别仍然立刻抢锁，迟滞只在同级之间生效。
参数默认含义
切换分差阈值0.15挑战者必须比当前锁定好出此比例（按半对角线归一化）才有资格抢锁。
连续优势帧数 (lock_switch_min_frames)3挑战者必须连续这么多帧保持分差优势。外部资料里的"切换目标延迟"指的就是这个，单位是帧——UI 滑条右侧实时显示 ≈ XX ms @ 当前捕获 fps。
锁定保持帧数 (lock_hold_min_frames)10锁定后强制保持目标的最少帧数；期间忽略所有切换条件，抑制抖枪 / 火焰 / 烟雾导致的乱切。

切换目标延迟的推荐区间

外部资料推荐 20 ~ 100 ms（太小乱锁，太大不锁）。换算成帧数：

捕获 fps20 ms50 ms100 ms
601 ~ 2 帧3 帧6 帧
1202 ~ 3 帧6 帧12 帧
2405 帧12 帧24 帧
UI 现在直接在滑条右侧打 ≈ XX ms @ N fps，不用心算。改捕获分辨率 / 捕获方式（UDP / TCP / OpenCV）会改变实际 fps，反过来影响这两个滑条的"实际 ms"——换捕获方式后回来确认下。

威胁度优先级

启发式给每个 track 算 [0,1] 的"威胁分"，乘进锁定打分：

  • 类别提示：head > person > 其他。
  • 运动方向：朝准星运动 → 高分；背离 → 低分。
  • 历史置信：被持续追踪很久的 track 比新冒出来的更可信。
参数默认含义
threat_priority_enabledoff主开关。
threat_weight0.50权重 [0,1]。0=完全忽略，1=高威胁打分变 0、低威胁打分翻倍。
threat_head_class_id-1告诉启发式哪个类别算 head。-1 表示未指定。
threat_body_class_id-1告诉启发式哪个类别算 body / person。-1 表示未指定。
威胁度不是新模型，而是生产启发式：类别层级 + 目标相对准星的运动方向 + 追踪命中帧数共同影响锁定排名。threat_weight=0 等价于关闭这部分乘子。

智能扳机（自动开火）

是什么
满足"准星在命中区 + 枪口稳定"时自动按下左键，保持指定时长后释放。
为什么有
把"瞄到位才开火"这个判断从人脑卸到代码上。比无脑 trigger 更克制、更像人。
影响
直接通过当前输入设备（WIN32 / KMBOX / MAKCU / GHUB / Arduino）按下左键，不是只点亮 UI。

三个开火门

  1. 命中概率门：pivot 到准星距离 ≤ bbox 短边 × hit_radius_frac 的范围内。
  2. 稳定门：最近 N 帧鼠标位移 RMS ≤ variance_max_px。
  3. 冷却门：上一发释放后过了至少同等长度的时间。

三门同时开 → 按下左键，保持 fire_duration_ms，到点释放。

参数默认含义
smart_trigger_enabledoff总开关。
hit_radius_frac0.55命中区 = bbox 短边 × 此值 / 2。0.55 ≈ 留 45% 安全边。
variance_max_px6.0最近 N 帧鼠标 RMS 位移上限，单位检测像素。超过 = 还在甩枪不开火。
window_frames8RMS 窗口帧数。
min_prob0.70命中概率门槛 [0,1]。
fire_duration_ms40每次按下保持多久。单发 40ms · 长按喷射 200ms+。释放后强制同等冷却（占空比 ≤ 50%）。

五重防护：按键绝不卡住

  1. 到点正常释放（PID 步内）。
  2. 检测断流 → checkAndResetPredictions watchdog 兜底释放。
  3. 松开瞄准热键 → clearQueuedMoves → 强制释放。
  4. 切换热键 / 改参数 → updateParams → 强制释放。
  5. 会话结束 / 析构 → 第一行就释放。

调参步骤

  1. 所有其他模块都稳定后再开扳机。前面任何一项不稳，扳机只会越打越错。
  2. 从默认值起步。先开站桩目标看是否能稳定单点。
  3. 如果错过该开火的窗口：降 min_prob 到 0.55，或提 hit_radius_frac 到 0.65。
  4. 如果在不该开火时也开（比如目标穿越准星但其实瞄的是另一目标）：提 min_prob 到 0.80，或降 variance_max_px 到 4.0。
  5. fire_duration_ms：手枪/单发武器 30~50；连射武器 150~250。
fire_duration_ms 太短（<20ms）某些游戏可能识别不到点击；太长（>500ms）冷却也跟着延长，节奏会很怪。

准星找色

游戏里的准星不一定在屏幕几何中心（开镜偏移、抖枪、后坐力补偿）。开启后系统会找屏幕中心 ROI 内符合 HSV 颜色的小色块作为动态准星，PID 误差以这个色块位置为参考点。

参数默认含义
crosshair_rect_w / h40 / 40从屏幕中心抠多大的方框找色（像素）。
crosshair_min_pixel_count4ROI 内符合 HSV 的像素数下限。低于此值视为没找到——抑制噪声 / 单像素误命中。
crosshair_close_radius1MORPH_CLOSE 的核半径（像素）。0 = 关闭；1~3 适合渐变 / 中心带白点的准星样式；>5 容易把红色噪点合到一起。
颜色 HSV 段红色双段HSV 色相在 0/180 处会回环，纯红色需要两段（0~10 + 160~179）。每段独立 enabled，便于在面板里临时关掉一段做实验。
每热键开关offcrosshair_detect_enabled 在每个 hotkey 里独立打开；调色板和 ROI 是全局共享的。

深度推理 + 抑制占比

是什么
用 Depth Anything 估每个像素的深度，把"远端/被遮挡"区域生成一张二值遮罩。落在遮罩里的 detection 直接丢。
为什么有
解决"识别框压在墙上 / 远处贴图被误识别"。
影响
仅 TRT 后端可用。增加约 2~5ms GPU 时间。
参数默认含义
depth_inference_enabledtrue主开关。
depth_mask_enabledfalse是否真正用遮罩抑制 detection（仅 enable 推理只是显示用）。
depth_mask_near_percent20把深度直方图的多少百分比像素视为"近端"。
depth_mask_invertfalse反转——某些游戏深度方向相反时切换。
depth_mask_expand0遮罩做形态学膨胀的核半径。
depth_mask_hold_frames0遮罩抑制保持帧数。0 = 跟当前帧；>0 给被遮罩像素 N 帧 TTL。
depth_mask_suppression_ratio0.30bbox 内被遮罩覆盖的像素占比必须 ≥ 此值才丢弃。0=任何 1 像素命中即丢（旧行为，易抖）；1=永不丢。

调参步骤

  1. 开 depth_inference_enabled 但先不开 mask，看深度图叠层是否合理。
  2. 合理后开 depth_mask_enabled，调 near_percent 让墙/天空被遮罩、玩家不被遮罩。
  3. 如果发现玩家偶尔被深度抹掉一帧导致丢锁：把 suppression_ratio 从 0.30 提到 0.50。
  4. 如果墙边 detection 跳来跳去：把 suppression_ratio 提到 0.40 + hold_frames 设 5。

输入设备

六选一：WIN32 / GHUB / ARDUINO / KMBOX_NET / KMBOX_A / MAKCU。隐蔽性顺序大致是 KMBOX/MAKCU > ARDUINO > GHUB > WIN32。

  • WIN32：直接 SendInput，0 硬件成本，但容易被反作弊识别。
  • GHUB：调罗技驱动 DLL，比 WIN32 略好。
  • ARDUINO：通过串口让 Arduino 模拟 USB HID。需要硬件。
  • KMBOX_NET / KMBOX_A：KMBox 系列硬件设备，网络 / USB-A 两种连接方式。
  • MAKCU：另一种硬件 mouse-injector。
参数默认含义
arduino_port / baudrateCOM0 / 115200Arduino 串口与波特率。
arduino_16_bit_mousefalse启用 16-bit 鼠标位移协议。只有固件也支持时才打开。
arduino_enable_keysfalse允许 Arduino 路径派发按键。智能扳机需要硬件侧支持左键按下/释放。
kmbox_net_ip / port / uuid10.42.42.42 / 1984 / DEADC0DEKMBOX_NET 连接参数。
kmbox_a_pidvid空KMBOX_A USB 设备匹配信息。多设备时用于指定目标硬件。
makcu_port / baudrateCOM0 / 115200MAKCU 串口与波特率。
所有"按键"操作（包括智能扳机的左键自动开火）都统一通过当前选中的输入设备派发。换设备无需改任何应用层逻辑。

定时Sleep(ms) / GetRunningTime()启动时调过 timeBeginPeriod(1),Sleep(7) 是真 7 ms 而不是 round 到 15.6 ms。Sleep 可被 reload / unload 中断。
查询GetMousePosition() / IsKeyDown(name_or_vk)返回当前光标坐标 / 任意键状态。
日志OutputLogMessage(fmt, ...) / OutputDebugMessage / ClearLogprintf 风格(走 Lua string.format),输出到"宏脚本"页的日志面板。
开关EnablePrimaryMouseButtonEvents(bool)动态开/关 LMB 事件派发,优先级高于 config。
兼容桩PlayMacro(name)查 _G[name],如果是函数则同步调用(无参无返回)。常见用法 = 脚本顶层定义同名函数当宏。
AbortMacro / SetBacklightColor / RestoreBacklightColor / SetMKeyState / GetMKeyState / GetDate原 LGS 硬件 API。我们没 LED / 没 M-keys / 单线程,这些都是 no-op 或最小实现,不会报错。

事件名 + 脚本骨架

当前实现派发的事件:

事件名arg触发时机
MOUSE_BUTTON_PRESSED1..5poller 检测到鼠标键边沿(2 ms 轮询)。arg=1 受 macro_primary_button_events 节流。
MOUSE_BUTTON_RELEASED1..5同上,松开。
PROFILE_ACTIVATED1脚本加载成功后立即触发(reload 也算)。
PROFILE_DEACTIVATED1主动 unload 时触发。

典型压枪脚本骨架:

EnablePrimaryMouseButtonEvents(true)

local recoil_pattern = {
    {0, 4}, {1, 6}, {-2, 5}, {-1, 7}, {0, 6},
}

function OnEvent(event, arg)
    if event == "MOUSE_BUTTON_PRESSED" and arg == 1 then
        for _, step in ipairs(recoil_pattern) do
            if not IsMouseButtonPressed(1) then break end
            MoveMouseRelative(step[1], step[2])
            Sleep(8)
        end
    end
end

线程模型

  • worker 线程:从事件队列取一个事件,锁住 Lua VM,调 OnEvent,等返回再取下一个。OnEvent 串行执行,新事件排队 —— 与 G HUB 行为一致。
  • poller 线程:2 ms 轮询五个鼠标键,检测到边沿就投递事件。
  • 合成抑制:脚本自己 PressMouseButton(1) 不会被 poller 当成物理按下回灌成事件,避免递归。
  • 取消机制:Lua 计数 hook(每 100 k 指令)+ 可中断 Sleep 双重检查 stop_requested / abort_pending,卡住的脚本能被 reload / unload 强行打断。
  • 定时器精度:启动时 timeBeginPeriod(1),关闭时 timeEndPeriod(1)。
左键事件默认不派发到 OnEvent —— 大多数 G HUB 脚本会在文件顶部写 EnablePrimaryMouseButtonEvents(true)。粘贴脚本后没反应先检查这一行,或者勾上 UI 里的"接收左键事件"。
不要在脚本里做不可中断的阻塞(网络请求、大文件读取、阻塞 IO):它们绕过我们的 hook 和 cancellable Sleep,reload 会等不到。需要异步行为时,把任务拆成 Sleep 间隔的小步。
PressAndReleaseMouseButton / PressAndReleaseKey 中间的 8 ms dwell 不是可中断 Sleep —— 这是为了避免 abort 时按键卡在按下状态。如果你的脚本依赖这条假设,知道就好。

瞄准轨迹回放（调试神器）

在调试面板开启"启用环形录制"，系统会把每帧的 detections + locked id + pivot + 热键状态推到一个环形缓冲（保留 N 秒）。点"快照并慢放"会浮出一个 ImGui 窗口，按 0.25x（或你设的速度）重放最近 N 秒：

  • 橙色框 = 当前帧的锁定目标
  • 蓝色框 = 其他检测
  • 红点 = pivot 锚点（PID 的目标点）
  • 红框边 = 该帧瞄准热键被按下

调 PID 时极其有用：你能直观看到鼠标"为什么过冲"、"为什么粘不住"——是 pivot 跳了一下，还是 PID 拉过头了。

参数默认含义
replay_record_enabledfalse启用环形录制。关闭时无任何分配。
replay_seconds10缓冲长度。10 秒大约够覆盖一次完整对枪。
replay_playback_speed0.250.25x 慢放最常用。

动态 FOV 实时读数

调试面板有个"动态 FOV (实时)"块，显示当前生效的 FOV 直径。开了动态 FOV 但读数始终是 0×0 = mouse 循环还没有跑过任何 tick（一般是没按热键）。读数比基线明显小很多 = 收缩生效中。
